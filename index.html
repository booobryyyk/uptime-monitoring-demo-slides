Monitoring Techniques

Synthetic Monitoring

Simulated requests at regular intervals from multiple locations. Proactive — catches outages before users report them.

Real User Monitoring

JavaScript on pages measures actual user experience (load time, errors, Core Web Vitals). Reactive — shows real-world impact.

Error Budget / SLO

Google SRE approach: define an SLO (e.g. 99.9%), track allowed downtime per quarter, alert when burn rate is too high.

Chaos Engineering

Netflix approach: deliberately inject failures (Chaos Monkey) to verify resilience before real incidents happen.

We use: Uptime Kuma for synthetic monitoring, Sentry for real user monitoring

Uptime Kuma Under the Hood

A self-hosted, open-source monitoring tool. An alternative to Pingdom or UptimeRobot — free, fully under our control, and extensible.

  • Node.js — naturally effective for I/O-bound tasks like HTTP health checks. Handles hundreds of monitors concurrently on a single process.
  • 90+ Notifications — PagerDuty, Slack, Telegram, email, webhooks — built-in integrations for alerting on status changes.
  • MariaDB on RDS — two storage options: SQLite (default, single-file) or MariaDB. We chose MariaDB for reliability and long-term heartbeat history storage.

Why Code-First Matters

ClickOps doesn’t scale

Adding 157 monitors by clicking through UI forms? No review, no history, no rollback.

Audit Trail

Git blame shows who added each monitor, when, and why. PR description has the full context.

Consistency

Templates enforce defaults (retry interval, notification type, tags) — no monitor is misconfigured.

Disaster Recovery

Uptime Kuma dies? Re-run all seeders on a fresh instance — back to full coverage.

What a Seeder Looks Like

import type { Knex } from 'knex';
import { createMonitor } from '../src/modules/monitors/templates/createMonitor';
import { deleteMonitor } from '../src/modules/monitors/templates/deleteMonitor';
import { NOTIFICATION_TYPE } from '../src/modules/notifications/notifications.constants';
import { TAG_NAME } from '../src/modules/tags/tags.constants';
import { deriveMonitorName } from '../src/modules/monitors/monitors.utils';

const MONITOR_URL = 'https://mate.academy/lpa/marathon/ui-ux-design';

export async function up(knex: Knex): Promise<void> {
  await createMonitor({
    knex,
    url: MONITOR_URL,
    name: deriveMonitorName(MONITOR_URL),
    notificationType: NOTIFICATION_TYPE.PagerDuty,
    tag: TAG_NAME.MateAcademy,
    active: true,
    interval: 300,            // 5 minutes
    retryInterval: 20,
  });
}

export async function down(knex: Knex): Promise<void> {
  await deleteMonitor({ knex, url: MONITOR_URL });
}

Real seeder from production. One file. Reviewable. Rollbackable.

Lessons Learned

  • Start with synthetic — covers the most ground with least effort
  • Code-first pays off quickly — no config drift, easy onboarding, reviewable changes
  • Monitoring finds surprises — exposed a memory leak we didn’t know about
  • Self-hosted = cost-effective — replaces pricy external tools with full control
